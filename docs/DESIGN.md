# StormFuse — Design Document

## 1. Purpose

StormFuse is a focused Windows desktop application that wraps `ffmpeg` / `ffprobe`
to perform exactly two workflows end-to-end:

1. **Combine** — join multiple video files (MKV and/or MP4) into a single output,
   using stream-copy concatenation when inputs are compatible and a normalize-then-concat
   pipeline when they are not.
2. **Compress** — re-encode a single input so the resulting file lands under a
   user-chosen size ceiling (default tuned to the **MFC Share 10 GB** per-file upload limit).

Both workflows prefer **NVIDIA NVENC** for re-encode work, fall back silently to
`libx264` on machines without a working NVENC path, and never re-encode when a
lossless stream-copy produces identical results.

The application is intentionally minimal: one window, two tabs, a few controls,
and exhaustive structured logging so that failures can be diagnosed by a human or
by another AI agent reading the log files.

```
┌────────────────────────────────────────────────────────────────┐
│                          PyQt6 UI                              │
│  MainWindow ─┬─ CombineTab ── FileList, RunControls            │
│              ├─ CompressTab ── InputPicker, SizeSlider, RunCtl │
│              └─ Menu: File, Help (About, Open Logs, Clear Logs)│

                           │ Qt signals/slots
┌──────────────────────────▼─────────────────────────────────────┐
│                        Job layer                               │
│   CombineJob, CompressJob  — QThread-based, cancellable        │
│   Emits: progress, log line, completion / failure              │
└──────────────────────────┬─────────────────────────────────────┘
                           │
┌──────────────────────────▼─────────────────────────────────────┐
│                      ffmpeg subsystem                          │
│   locator     : resolves bundled ffmpeg/ffprobe paths          │
│   probe       : ffprobe JSON wrapper → typed dataclasses       │
│   signatures  : compare probes for stream-copy eligibility     │
│   encoders    : NVENC detection + arg builders                 │
│   bitrate     : target-size → video bitrate math               │
│   concat      : stream-copy-or-normalize decision + planning   │
│   runner      : subprocess.Popen + `-progress` parser          │
└──────────────────────────┬─────────────────────────────────────┘
                           │
                     ffmpeg.exe / ffprobe.exe
```

No module outside `ffmpeg/` may spawn subprocesses, and nothing outside `ui/` may
import `PyQt6` widgets or GUI classes; `jobs/` may import `PyQt6.QtCore` for signals
and threading primitives. These layering rules are enforced by `pylint`'s
`import-outside-module` checks plus a small custom `ruff`/`tach`-style guard
described in §13.

## 5. Functional specification

### 5.1 Combine workflow

**Inputs.** A user-ordered list of MKV and/or MP4 files.

**Ordering.**
- On adding files: the list is initially sorted by **filename** (locale-aware,
  case-insensitive). This matches the PowerShell references' default behavior.
- An "Auto-sort by filename timestamp" button parses timestamps out of filenames
  using the same two patterns recognized by `combine_obs_v2.ps1`:
  - `YYYYMMDD-HHMMSS` anywhere in the basename (OBS convention)
  - `M-D-YYYY - HHMM[am|pm]` (MFC convention)
  Items lacking a recognizable timestamp are left in their current positions and
  flagged with a warning icon in the list.
- The user can drag rows to reorder, use Up/Down buttons, or remove rows.
- No hidden filename-based preset (no "Rin / Club Show" template).

**Stream-copy decision.** Before running, each input is probed. Stream-copy is
chosen if and only if **every** input matches on:

- `codec_name` (video)
- `width`
- `height`
- `pix_fmt`
- `r_frame_rate` (within ±0.01 fps tolerance)
- `codec_name` (audio, first stream)
- `sample_rate`
- `channels`

If any field mismatches, the **normalize-then-concat** strategy is used:
- Target format = the input with the largest pixel count, tiebreak by highest fps.
- Each non-matching input is re-encoded to an intermediate MKV in a temp dir
  (NVENC CQ 18 / libx264 CRF 18 — "visually lossless") with scaling + letterbox
  padding to preserve aspect ratio, then `fps=<target>`, `format=yuv420p`.
- Intermediates are stream-copy concatenated into the final output.
- Intermediates are **deleted on success**, **retained on failure** for debugging,
  and their location logged.

The decision and the reasoning behind it (which field mismatched on which file)
are always written to the log as a single structured event before the ffmpeg
invocation begins — so an AI agent reading the log can reconstruct why a
particular strategy was chosen.

**Output.** Default filename: `<first-input-basename>-combined.mkv` in the
directory of the first input. Editable in the UI before Run.

**Progress.** A single QProgressBar tracks the combined work:
- Stream-copy path: one progress phase, % = out_time / total_expected_duration
  (sum of input durations).
- Normalize path: N+1 progress phases (one per normalize + one concat),
  overall % = weighted sum by duration.

### 5.2 Compress workflow

**Input.** A single existing video file (MKV or MP4).

**Controls.**
- **Target size slider.** Range **1.0 GB – 10.0 GB**, step 0.1 GB, default **9.5 GB**.
  Tooltip explains that ffmpeg bitrate targeting is imprecise and that 9.5 GB is
  chosen to leave ~5% headroom under the MFC Share 10 GB ceiling.
- **2-pass toggle.** Off by default. When on:
  - NVENC: uses `-multipass fullres` (slower, tighter size hit).
  - libx264: runs a true two-pass encode (`-pass 1 -an -f null NUL` → `-pass 2`).
- **Audio.** Always AAC 192 kbps stereo 48 kHz (built-in `aac` encoder, LGPL,
  ships in every build). No `libfdk-aac`.
- **Container.** Always MP4 with `+faststart` for playback-friendly output.

**Bitrate math.**

```
overhead_fraction = 0.01              # 1% container reserve
target_bytes      = slider_gb * 1024**3
audio_bits_total  = 192_000 * duration_sec
video_bits_total  = target_bytes * 8 * (1 - overhead_fraction) - audio_bits_total
video_bitrate_k   = floor(video_bits_total / duration_sec / 1000)
maxrate_k         = floor(video_bitrate_k * 1.5)
bufsize_k         = floor(video_bitrate_k * 3.0)
```

If `video_bitrate_k <= 0` (target too small for audio + duration), the Run button
is disabled with an explanatory tooltip.

**Output.** Default filename: `<input-basename>-compressed.mp4` in the same directory.
Editable before Run.

### 5.3 NVENC detection & fallback

At application startup, a single probe runs:

1. `ffmpeg -hide_banner -encoders` is parsed for `h264_nvenc`.
2. If present, a **tiny 1-frame test encode** is run:
   `ffmpeg -f lavfi -i color=c=black:s=64x64:d=0.05 -c:v h264_nvenc -f null -`
3. If exit code 0, the app state is `encoder = NVENC_AVAILABLE`.
4. Otherwise `encoder = LIBX264_FALLBACK`. The status bar shows this plainly.
   The fallback reason (missing encoder / runtime error / stderr tail) is logged
   at INFO level; it is **not** a hard failure.

A "Re-check NVENC" button in the status-bar context menu lets the user re-probe
without restarting, useful when a driver update happens while the app is running.

### 5.4 Cancellation

Every job exposes a Cancel button while running. The cancel sequence:

1. Write `q\n` to ffmpeg's stdin and flush; close stdin. ffmpeg's documented
   graceful-quit signal. Wait up to **5 seconds** for exit.
2. On timeout: `process.terminate()` (Windows: `TerminateProcess`; Linux/CI: SIGTERM).
   Wait up to **3 seconds**.
3. On further timeout: `process.kill()`.
4. After the process exits, the job deletes:
   - The final output file (if any was created)
   - Any normalize-intermediate temp dir
   - Any 2-pass logfiles (`ffmpeg2pass-*.log`, `*.log.mbtree`)

The cancellation path is logged as a single structured event with the step that
actually terminated the process, so we can tell "graceful q" from "had to kill" later.

### 5.5 Single-job model

Only one job runs at a time. Starting a job disables the Run buttons on both tabs
and enables Cancel. No queue, no parallelism — this is intentional.

## 7. FFmpeg subsystem

### 7.1 `locator.py`

Resolves bundled binaries. Resolution order:

1. When running under PyInstaller: `sys._MEIPASS / "resources" / "ffmpeg" / "ffmpeg.exe"`.
2. When running from source: `<repo_root>/resources/ffmpeg/ffmpeg.exe` (this path
   is populated by `make installer-deps` or a dev-convenience script; it is
   `.gitignore`d).
3. **No PATH fallback.** The app intentionally does not use `ffmpeg` from PATH,
   because the point of bundling is reproducible behavior.

If the binaries cannot be found at startup, a modal error dialog explains the
install is corrupted and offers to open the repo's troubleshooting section.

### 7.2 `probe.py`

Thin wrapper over `ffprobe -v error -print_format json -show_streams -show_format`.
Returns a `FileProbe` dataclass with fully typed fields (`video: VideoStream | None`,
`audio: AudioStream | None`, `duration_sec: float`, `size_bytes: int`, raw `dict`
kept for debugging).

All ffprobe invocations include `--` before the filename and pass the filename as
a single argument (no shell). On non-zero exit, raises `ProbeError(path, stderr_tail)`.

### 7.3 `signatures.py`

```
@dataclass(frozen=True)
class VideoSignature:
    codec: str
    width: int
    height: int
    pix_fmt: str
    fps: float

@dataclass(frozen=True)
class AudioSignature:
    codec: str
    sample_rate: int
    channels: int

def signatures_match(a: FileProbe, b: FileProbe, *, fps_tolerance: float = 0.01) -> bool: ...
def describe_mismatch(a: FileProbe, b: FileProbe) -> list[str]: ...   # for logs/UI
```

### 7.4 `encoders.py`

- `detect_encoder(ffmpeg_path) -> EncoderChoice` runs the §5.3 probe.
- `compressed_video_args(choice, bitrate_k, two_pass: bool) -> list[str]`
- `normalize_video_args(choice) -> list[str]`  (CQ 18 / CRF 18)
- `audio_args() -> list[str]`   (always `-c:a aac -b:a 192k -ar 48000 -ac 2`)

Arg builders return **lists of strings**, never shell-escaped strings. The runner
never sees `shell=True`. This is also unit-test-friendly on Linux.

### 7.5 `bitrate.py`

Pure functions implementing §5.2 math. 100% unit-testable on Linux.

### 7.6 `concat.py`

Given an ordered list of `FileProbe`, decides:

- `ConcatPlan.stream_copy(inputs)` — all signatures match.
- `ConcatPlan.normalize_then_concat(inputs, target_sig, normalize_indices)` —
  includes which specific inputs need re-encoding and which could be copied.

The plan is serialized into the log so that an agent debugging a failure can see
exactly what was attempted.

### 7.7 `runner.py`

Wraps `subprocess.Popen` with:

- `stdout=DEVNULL`, `stderr=PIPE` (captured, tail kept for error reports).
- `stdin=PIPE` (required for graceful cancel).
- `-progress <pipe>` + `-nostats`; progress pipe is read in a dedicated thread
  and parsed into `ProgressEvent(out_time_sec, speed, bitrate_kbps, frame)`.
- `-y` (overwrite), `-hide_banner`.
- No `shell=True`, ever. All args are a list.
- On Windows, `creationflags=CREATE_NO_WINDOW` so no console pops up behind the UI.

The runner emits three Qt signals: `progress(ProgressEvent)`, `log_line(str)`,
and `finished(RunResult)` where `RunResult` includes exit_code, duration,
stderr_tail (last 200 lines), and the full argv actually invoked.

## 8. Job layer

```
class Job(QObject):
    progress = Signal(float, str)     # 0..1, phase label
    log      = Signal(dict)           # structured log event (also written to file)
    done     = Signal(JobResult)
    failed   = Signal(JobError)

    def run(self) -> None: ...
    def cancel(self) -> None: ...     # idempotent
```

`CombineJob` drives §5.1; `CompressJob` drives §5.2. Both are executed on a
`QThread`, never on the UI thread. Both expose a `job_id` (UUID7-ish monotonic)
that is stamped on every log line emitted during that job, making logs trivially
filterable.

## 9. Logging

Logging is a first-class feature of StormFuse, designed so that a human or an
LLM reading the log files can diagnose any failure without access to the
running system.

### 9.1 Log location & rotation

- Root: `%LOCALAPPDATA%\StormFuse\logs\`.
- Per-session file: `stormfuse-YYYYMMDD-HHMMSS-<shortpid>.log`.
- On app start, a `latest.log` file is written in parallel by a second handler attached to the same root logger (regular file, not a symlink; Windows symlinks need elevation).
- Retention: keep the most recent **20** session files; older files deleted at startup.

### 9.2 Format — structured JSON Lines + human mirror

The primary log stream is **JSON Lines** (one JSON object per line). Every record has:

```json
{
  "ts": "2026-04-23T14:05:22.123Z",
  "level": "INFO",
  "logger": "ffmpeg.runner",
  "event": "ffmpeg.start",
  "job_id": "combine-7f3c...",
  "msg": "Starting ffmpeg for concat",
  "ctx": {
    "argv": ["<ffmpeg>", "-y", "-progress", "<pipe>", "-f", "concat", "-i", "..."],
    "cwd": "C:\\Users\\...\\videos",
    "encoder": "stream_copy"
  }
}
```

Required fields: `ts`, `level`, `logger`, `event`, `msg`. Optional but conventional:
`job_id`, `ctx`, `error` (with `type`, `message`, `traceback`), `phase`, `pct`.

The **human mirror** is a derived, line-per-record textual view used for the in-app
log pane and for copy-paste into issues:

```
14:05:22.123 INFO  ffmpeg.runner  [combine-7f3c] ffmpeg.start  encoder=stream_copy
```

It is produced by the same `LogRecord`; both formats are emitted by the same
handler pair to keep them in lockstep.

### 9.3 Event catalog (non-exhaustive — must be implemented)

| Event | Level | When |
|-------|-------|------|
| `app.start` / `app.exit` | INFO | app lifecycle, includes version + platform |
| `nvenc.probe` | INFO | NVENC detection result incl. reason on fallback |
| `probe.start` / `probe.result` / `probe.error` | DEBUG / DEBUG / ERROR | per ffprobe call |
| `concat.decision` | INFO | carries `strategy`, mismatched fields, target sig |
| `job.start` / `job.finish` / `job.cancel` / `job.fail` | INFO / INFO / WARNING / ERROR | job lifecycle |
| `ffmpeg.start` | INFO | includes full argv |
| `ffmpeg.progress` | DEBUG | sampled at most 1/sec |
| `ffmpeg.stderr` | DEBUG | per non-empty line (kept for failure postmortems) |
| `ffmpeg.exit` | INFO or ERROR | exit_code, stderr tail on failure |
| `logs.clear` | INFO | user invoked Clear Log Files |

Guidance for contributors: **every user-visible error must have a preceding
structured event that names the root cause**. UI error dialogs quote the
`event` + `msg` + 10 lines of stderr tail, plus a "Copy diagnostic" button that
bundles `latest.log` + app version + OS + encoder state onto the clipboard.

### 9.4 Clear Log Files

Implementation:

1. Enumerate `%LOCALAPPDATA%\StormFuse\logs\*`.
2. For the file currently held by the logging handler (the active session log
   and `latest.log`): `handler.stream.seek(0); handler.stream.truncate(); handler.stream.flush()`.
   This zeroes the content without breaking the open handle.
3. For every other file: `os.unlink()`.
4. `PermissionError` on any individual file is caught and logged (at
   `logs.clear.partial`); the operation continues.
5. On completion, emit `logs.clear` with counts (deleted, truncated, failed).

## 10. Error handling philosophy

- Fail fast with clear messages. Never silently continue on a recoverable-but-noteworthy condition — log it and surface it.
- Exceptions thrown to the UI thread are caught by a single top-level handler
  that logs `app.unhandled` with the full traceback and shows a dialog with the
  "Copy diagnostic" button.
- User-facing messages name **what failed**, **why** (if known), and **what to try next**.
  Example: "Concat failed: input 3 has a different frame rate (29.97 vs 60). Try
  the 'Normalize then concat' strategy, or re-record input 3 at 60 fps."
- No stack traces in user-facing dialogs; they go to the log.

## 11. Tests

### 11.1 Layout

```
tests/
  unit/             # runs anywhere (Linux OK). Target: every PR.
  functional/       # requires Windows + bundled ffmpeg. Skipped off-Windows.
  fixtures/
    generate_media.py   # creates tiny test videos at CI time, not committed
```

### 11.2 Unit tests (Linux-compatible)

Mandatory coverage:

- `bitrate.py` — every branch of the math, incl. the "target too small" case.
- `signatures.py` — match/mismatch per field, fps tolerance edge cases.
- `concat.py` — strategy decisions for curated sets of synthetic `FileProbe`s.
- `encoders.py` — arg builders produce exact expected argv for each (encoder, 2-pass) pair.
- Filename timestamp parser — both supported formats + rejection of ambiguous inputs.
- JSON-log formatter — round-trips through `json.loads`, contains required keys.
- `runner.py` argv construction (without actually spawning ffmpeg) — via a mock
  `Popen` that captures the call.
- Signature dataclasses — frozen, hashable, equality semantics.

**No unit test spawns a real subprocess.** All ffmpeg/ffprobe interactions in
unit tests go through injected fakes.

### 11.3 Functional tests (Windows-only)

Marked `@pytest.mark.windows_only`. A top-level `conftest.py` applies
`pytest.skip("requires Windows")` automatically when `sys.platform != "win32"`,
so running `pytest` on Linux simply skips them.

Functional fixtures generate short (~1-2 second) test media via ffmpeg's built-in
synthetic sources at test setup time (`testsrc2`, `sine`), in multiple
resolutions and codecs, so tests don't ship binary media fixtures.

Functional coverage:

- End-to-end combine, stream-copy path (all matching inputs).
- End-to-end combine, normalize path (mismatched resolutions / fps).
- End-to-end compress, libx264 single-pass (GPU-independent).
- End-to-end compress, libx264 two-pass.
- NVENC-dependent tests (`@pytest.mark.requires_nvenc`) skip when NVENC probe
  fails on the runner, so CI without a GPU stays green.
- Cancellation: start a long job, cancel after 500 ms, assert output file is
  gone and job state is `cancelled`.
- Clear Log Files: create held + unheld files, run action, assert held files
  are truncated and unheld files deleted.

### 11.4 UI tests

Light, not exhaustive. `pytest-qt` drives:

- Main window opens, both tabs render.
- Adding files to Combine populates the list, sort buttons reorder correctly.
- Slider value updates the bitrate preview.
- About dialog contains the required attribution strings.

## 12. Build, packaging, distribution

### 12.1 Local developer build

- `make venv` creates `.venv` with Python 3.12.
- `make deps` installs `-r requirements-dev.txt` (which transitively pulls in
  `requirements.txt`).
- `make run` launches the app from source using the bundled ffmpeg under
  `resources/ffmpeg/` if present, or prompts to run `make fetch-ffmpeg`.
- `make fetch-ffmpeg` downloads the pinned gyan.dev essentials build, verifies
  SHA-256 against `build/ffmpeg.sha256`, extracts `ffmpeg.exe` and `ffprobe.exe`
  into `resources/ffmpeg/`. The path is `.gitignore`d.

### 12.2 Installer pipeline

Phase 1 — PyInstaller (onedir, not onefile; faster startup, better AV friendliness):

```
pyinstaller \
    --noconfirm \
    --name StormFuse \
    --windowed \
    --icon resources/icons/stormfuse.ico \
    --add-data "resources/licenses;resources/licenses" \
    --add-binary "resources/ffmpeg/ffmpeg.exe;resources/ffmpeg" \
    --add-binary "resources/ffmpeg/ffprobe.exe;resources/ffmpeg" \
    src/stormfuse/__main__.py
```

Phase 2 — NSIS (`build/installer/stormfuse.nsi`):

- Installs under `%ProgramFiles%\StormFuse` by default; per-user install supported
  via the NSIS `MultiUser` plugin.
- Creates Start Menu shortcut, optional Desktop shortcut.
- Writes `HKLM\Software\StormFuse` with `Version` and `InstallDir`.
- Registers Add/Remove Programs entry with publisher "Winds of Storm".
- Includes a license page that shows `LICENSE` (GPL v3).
- Uninstaller removes everything it installed; **does not** touch
  `%LOCALAPPDATA%\StormFuse` (logs/settings) unless user checks "Remove
  application data" on the uninstall wizard.

### 12.3 Code signing

Out of scope for v1. Design leaves a hook in the NSIS script and the GHA
workflow for `signtool` to be added later when a cert is available.

## 13. Linting, typing, style

All three of the following must pass with zero warnings before merge:

- **ruff** — project config in `pyproject.toml`; selected rulesets: E, F, W, I,
  B, UP, SIM, RUF, PL (selected). Line length 100. Targets `py312`.
- **mypy** — `--strict` mode for `src/stormfuse/`. `tests/` may use `--strict-optional`
  but relaxed everywhere else. Qt types come from the official PyQt6 stubs.
- **pylint** — with a project `.pylintrc` that disables rules duplicated by ruff
  (to avoid double-reporting) but keeps pylint's design checks
  (too-many-args, too-many-branches, cyclic-import).

Import-layering rule: `pylint`'s custom plugin forbids `stormfuse.ui` from being
imported by `stormfuse.ffmpeg` or `stormfuse.jobs`, and forbids `subprocess` in
any module outside `stormfuse.ffmpeg`. Enforced in CI.

Formatting is handled by **ruff format** (no black). Imports sorted by ruff's `I`.

## 15. Makefile targets

| Target | Action |
|--------|--------|
| `make venv` | Create `.venv/` with Python 3.12 (fails clearly if absent) |
| `make deps` | Activate venv and `pip install -r requirements-dev.txt` |
| `make fetch-ffmpeg` | Download + SHA-256 verify + extract ffmpeg into `resources/ffmpeg/` |
| `make run` | Launch app from source |
| `make lint` | Run ruff, mypy, pylint in that order; fail on any |
| `make format` | Run `ruff format` + `ruff check --fix` |
| `make test` | Run unit tests only (default; works on Linux) |
| `make test-functional` | Run functional tests (Windows; skips elsewhere) |
| `make test-all` | Unit + functional |
| `make installer` | PyInstaller → NSIS → `dist/StormFuse-Setup-<ver>.exe` |
| `make clean` | Remove `dist/`, `.pytest_cache/`, `__pycache__/`, `.venv/`, and generated artifacts under `build/`; keep tracked build sources (`.py`, `.nsi`, `.sha256`, `.spec`) |

`requirements.txt` = runtime deps (`PyQt6`, minimal stdlib-plus).
`requirements-dev.txt` starts with `-r requirements.txt` and adds
`pytest`, `pytest-qt`, `pytest-cov`, `ruff`, `mypy`, `pylint`, `PyInstaller`,
plus any type stubs.

## 16. GitHub Actions workflows

### 16.1 `ci.yml`

- Triggers: `push`, `pull_request`.
- Runs on `ubuntu-latest`.
- Branch filters: `master` for both `push` and `pull_request`.
- Permissions: `contents: read`.
- Steps: checkout → setup Python 3.12 → `make deps` → `make lint` →
  `make generate-third-party` → `make test`.
- Uploads `coverage.xml` as an artifact.

### 16.2 `release.yml`

- Triggers: `push` tags matching `v*`, and `workflow_dispatch` with an optional
  `tag` input for manual rebuilds of an existing tag.
- Uses `TAG_NAME = inputs.tag || github.ref_name` and checks out
  `refs/tags/${TAG_NAME}` in every job, so rebuilds always use the tagged source.
- Permissions: `contents: write`.
- Jobs:
  1. `lint-and-test` on `ubuntu-latest`: checkout tagged source → setup Python 3.12
     → `make deps` → `make lint` → `make generate-third-party` → `make test`
     → upload `coverage.xml`.
  2. `build-installer` on `windows-latest`: checkout tagged source → setup Python 3.12
     → `make deps` → `make fetch-ffmpeg` (validates pinned SHA-256 against the
     configured archive filename, not any redirected CDN UUID path) →
     `make generate-third-party` → `make test` → `make installer` →
     upload the installer artifact.
  3. `release` on `ubuntu-latest`: download the installer artifact and publish the
     GitHub release with generated notes via `softprops/action-gh-release`.

## 19. Explicit non-goals (v1)

- No batch queue / multi-job parallelism.
- No video preview / scrubber UI.
- No re-mux to formats other than MKV (combine) / MP4 (compress).
- No HDR-aware handling; SDR only.
- No audio-only mode.
- No 2-pass NVENC for normalize intermediates (quality-targeted CQ is sufficient).
- No auto-update / telemetry.
- No code signing (hook exists, cert does not).
- No installer localization beyond English.
